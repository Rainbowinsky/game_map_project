# 第二阶段实施计划

## 1. 工作包与依赖

| 工作包              | 交付物                                                     | 前置条件                | 验收重点                                             |
| ------------------- | ---------------------------------------------------------- | ----------------------- | ---------------------------------------------------- |
| P2-0 契约冻结       | v2 migration、对象/主题/location schema、fixture、迁移测试 | 第一阶段 P12 收口已合入 | 旧图章可读写；非法几何、跨地图引用和 v1 写入均被拒绝 |
| P2-1 主题基础       | ThemeRegistry、原创亮色主题、渲染/导出 token 注入          | P2-0                    | 编辑器与 PNG 导出使用同色板；未知主题可恢复          |
| P2-2 路径与区域     | path/region object、工具、编辑节点、命令、投影             | P2-0、P2-1              | 道路/河流/区域可创建、编辑、撤销、保存和刷新恢复     |
| P2-3 地形笔刷       | terrain-stroke、预览、抽样、橡皮擦策略                     | P2-0、P2-1              | 一次笔势一条历史/保存批；长笔势和超限输入安全失败    |
| P2-4 文字与地点     | text/marker、Location CRUD、详情面板、搜索与定位           | P2-0、P2-1              | 地点与标记原子保存；文本无 HTML 注入                 |
| P2-5 素材库         | multipart 上传、图片校验/缩略图、分类、引用安全删除        | P2-0；图像依赖选型通过  | 路径/MIME/SVG 攻击被拒绝；跨用户资源隔离             |
| P2-6 缩略导航与收口 | minimap、可访问性、性能观测、E2E、文档                     | P2-2 至 P2-5            | 点击定位正确；所有一、二阶段回归门禁通过             |

P2-2 与 P2-3 在 P2-0/P2-1 后可并行开发；P2-4 的纯文字部分可提前，但 marker 与地点事务依赖 P2-0。P2-5 独立于画布工具，但上传的 asset 不能在 schema/引用检查完成前接入对象。



## 2. 每包实施清单

### P2-0：先冻结契约

1. 为新对象、路径节点、笔刷、主题、地点及 API DTO 添加 strict Zod schema 和正反 fixtures。
2. 扩展 `MapObject`、operation、repository mapper 与 Prisma migration；实现 1 → 2 文档迁移。
3. 给 location/marker 关系、asset 引用和跨地图引用写集成测试。
4. 不创建 UI 工具；这一步的结果应是可验证且向后兼容的写入内核。

### P2-1：主题作为唯一视觉入口

1. 把 `map-artwork.ts`、网格与选择叠层中的颜色迁移到解析后的 token。
2. 让 `ObjectProjection` 与 PNG exporter 接收相同主题，不改变已有图章的渲染语义。
3. 为主题定义、回退与导出一致性增加单元测试。

### P2-2/P2-3：几何绘制

1. 先实现命令、预览、pointer 生命周期和几何测试，再接入工具栏与属性面板。
2. 路径编辑按绘图一样点击，开始绘制，再点击取消绘制
3. 笔刷按世界距离重采样；橡皮擦第一版删除相交的 stroke 整体，不实现破碎重组，并在 UI 中明确提示。
4. 每次提交均走 command → patch bus → operation journal → autosave → renderer 增量投影链路。

### P2-4/P2-5：资料与素材

1. 地点表单先以纯文本字段实现，搜索按名称、类型和标签；点击 marker 打开详情并可定位。
2. 上传文件先落临时 key，验证、哈希和缩略图均成功后才建立 Asset 记录并移动到最终 UUID key。
3. 对素材删除实施“有引用即拒绝”的保守策略；清理 worker/队列留待第四阶段，不假装异步任务已经存在。
4. 自定义素材只经资产 ID 被引用，前端永不接收或拼接磁盘路径。

### P2-6：可用性与验收

1. 缩略图用独立投影更新，并加键盘可达的定位控件与 aria 标签。
2. 补足 500/2,000 几何对象的观察场景；只记录可复现基准，不在 CI 设跨机器 FPS 阈值。
3. 更新 README、API 契约、素材许可清单与本阶段实施报告。

## 3. 测试矩阵

| 范围             | 单元/契约                                           | 集成                                 | 端到端                             |
| ---------------- | --------------------------------------------------- | ------------------------------------ | ---------------------------------- |
| schema/migration | 边界、JSON 字节、几何、自交、v1 → v2                | 旧地图加载和 operation 原子性        | 刷新后恢复已有地图                 |
| 路径/区域/笔刷   | 抽样、命中、Bezier、撤销合并                        | revision、锁定图层、Chunk 计数       | 绘制、编辑、撤销、刷新             |
| 文字/地点        | 文本/缩放范围/marker 引用                           | 归属、跨地图与删除回滚               | 创建地点、点击标记、搜索定位       |
| 资产             | MIME、magic bytes、SVG sanitizer、hash、分类 schema | 用户隔离、分页、引用删除、无路径泄露 | 上传、选用、刷新后使用             |
| 主题/缩略图      | token 回退、坐标转换                                | 无需新增数据库断言                   | 主题切换、导出一致、缩略图拖动定位 |

每次工作包合并前运行：

```powershell
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

涉及 API、Prisma 或 storage 时加跑 `pnpm test:integration`；UI 主路径变更加跑 `pnpm test:e2e`。P2-6 发布候选还要完整运行两者和既有 `pnpm --filter @fantasy-map/web test:benchmark`。

## 4. 开工前决策门

在 P2-0 结束、P2-5 开始前由项目负责人确认以下两项，避免无意引入重型或不安全依赖：

1. 服务端图像处理库：必须支持 PNG/JPEG/WebP 尺寸读取与缩略图，给出维护状态、许可和 Windows/CI 安装验证；SVG 单独使用受限 sanitizer，不将其交给通用 XML 解析后直接服务。
2. 主题产品边界：P2 交付内置主题切换还是允许用户编辑并保存主题。默认采用前者；后者需要新增 Theme 持久化、所有权、版本和资源引用设计，超出本计划。

## 5. 风险登记

| 风险                                 | 控制措施                                                         |
| ------------------------------------ | ---------------------------------------------------------------- |
| 任意 JSON 破坏 renderer 或保存兼容性 | 新对象使用 discriminated union、严格 update 白名单、版本迁移测试 |
| 路径/笔刷点过多导致保存或渲染卡顿    | 采样、点数与字节上限；提交前简化；预览与持久化分离               |
| 上传绕过类型校验或泄露路径           | magic bytes/尺寸/SVG sanitizer、UUID key、受控 URL、日志脱敏     |
| 地点和 marker 写入不一致             | 同一 revision 事务、跨引用预验证、回滚测试                       |
| 主题值继续散落造成导出不一致         | P2-1 先建 registry；renderer/exporter 共享 token                 |
| 第二阶段提前承诺大地图性能           | 明确维持全量加载边界，将流式 Chunk/索引/LOD 留给第三阶段         |
