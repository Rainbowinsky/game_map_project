你是一名资深全栈工程师、WebGL 图形开发工程师和地图编辑器架构师。请设计并逐步开发一款 Web 端世界观地图绘制工具。

一、项目定位

开发一款面向小说作者、游戏策划、跑团主持人和世界观创作者的 2D 世界地图编辑器。

产品需要支持：

1. 创建超大尺寸的虚构世界地图。
2. 绘制大陆、岛屿、海洋、湖泊、河流、山脉、森林、道路和区域。
3. 放置城镇、遗迹、营地、传送点等地图图章。
4. 创建国家、地区、地点和世界观资料。
5. 对地图内容进行分层编辑。
6. 保存为可继续编辑的工程。
7. 导出 PNG、WebP 和项目文件。
8. 后续能够扩展桌面端、多人协作和在线互动地图。

视觉方向参考开放世界奇幻游戏地图的设计语言，例如：

- 清晰柔和的幻想世界地图配色。
- 羊皮纸、水彩、手绘等质感。
- 风格化的山脉、森林、城镇和地标图章。
- 柔和地形渐变、海岸描边和地形阴影。
- 不同缩放等级展示不同密度的内容。
- 简洁、易辨识的图标和地名排版。

只参考总体设计语言，不复制任何现有游戏的具体地图、图标、纹理、角色、Logo、字体或其他受版权保护的资源。所有默认素材应为原创、程序生成或使用许可证明确的资源。

采用前后端分离架构。

- 地图实时绘制。
- 编辑器交互。
- 选区与变换。
- 撤销和重做。
- 前端临时状态。
- 视口裁剪。
- 自动保存缓冲。
- 低分辨率预览导出。

后端负责：

- 用户和项目管理。
- 地图文档保存。
- 地图版本历史。
- 资源上传与管理。
- 权限验证。
- 大型导出任务。
- 项目文件打包。
- 后续瓦片生成与多人同步。

MySQL 是长期数据的真实来源。

Redis 只用于：

- 缓存。
- 后台任务队列。
- 导出进度。
- 限流。
- 临时在线状态。
- 后续多人协作消息。

不要把 Redis 当作地图工程的唯一存储。

七、超大地图设计

地图可能达到数万或数十万世界单位，不能把整个地图作为一张超大 Canvas 或纹理加载。

必须采用以下设计：

1. 使用自定义二维世界坐标，不强制采用真实经纬度。
2. 将世界划分为固定尺寸 Chunk，例如 512×512 或 1024×1024 世界单位。
3. 仅加载当前视口及周边缓冲区内的 Chunk。
4. 离开视口的资源应暂停渲染、卸载或进入缓存。
5. 使用空间索引查询视口中的对象。
6. 图章使用实例对象保存，不能将整张地图烘焙成一张图片。
7. 对象密集时启用批处理、纹理图集和对象池。
8. 根据缩放等级使用 LOD：
   - 远距离只显示大陆、国家和主要城市。
   - 中距离显示地区、城镇和主要道路。
   - 近距离显示森林、遗迹和次要标记。
9. 复杂几何计算放入 Web Worker。
10. 大地图保存采用增量保存，不在每次修改时上传整个文档。
11. 后期使用 MapLibre 或自研瓦片管理器加载发布后的地图瓦片。
12. 地图导出使用分块渲染和拼接，避免超过浏览器最大纹理尺寸。

建立以下核心结构：

interface WorldPoint {
  x: number;
  y: number;
}

interface CameraState {
  x: number;
  y: number;
  zoom: number;
}

interface ChunkCoordinate {
  x: number;
  y: number;
}

interface MapObject {
  id: string;
  type: string;
  layerId: string;
  chunkId: string;
  x: number;
  y: number;
  rotation: number;
  scaleX: number;
  scaleY: number;
  visible: boolean;
  locked: boolean;
  metadata?: Record<string, unknown>;
}

八、地图文档设计

定义具有版本号的地图文档格式：

interface MapDocument {
  schemaVersion: number;
  id: string;
  projectId: string;
  name: string;
  width: number;
  height: number;
  themeId: string;
  background: MapBackground;
  layers: MapLayer[];
  settings: MapSettings;
  createdAt: string;
  updatedAt: string;
}

大型对象数据不要全部永久嵌套在一个巨大 JSON 中。

建议：

- MapDocument 保存地图元数据和图层结构。
- MapObject 按 Chunk 或图层分开保存。
- 地图操作增量保存。
- 发布时生成不可编辑的快照。
- 所有数据结构都使用 Zod 进行运行时验证。
- schemaVersion 用于未来数据迁移。

对地图高频编辑数据采取以下策略：

- 前端立即更新。
- 本地内存记录未保存状态。
- 经过防抖后批量调用保存接口。
- 后端使用事务写入变更。
- 定期生成完整版本快照。
- 不要在每次鼠标移动时写 MySQL。

十三、性能目标

在主流桌面浏览器中：

- 空地图拖动和缩放保持接近 60 FPS。
- 视口内数千图章时仍可流畅操作。
- 总地图对象数量可以达到十万级，但只渲染可见对象。
- 不因地图总尺寸创建同尺寸 Canvas。
- 初次进入地图不加载全部高清资源。
- 资源采用纹理图集和按需加载。
- 高频指针事件使用 requestAnimationFrame 合并。
- React 状态更新不能驱动每一帧 PixiJS 渲染。
- PixiJS 场景状态与 React UI 状态适当解耦。
- 大型保存采用增量变更。
- 导出任务不能阻塞主编辑接口。

十四、安全要求

- 数据库密码只放在后端环境变量。
- React 中不能包含 MySQL 或 Redis 连接信息。
- 对所有接口进行身份和权限检查。
- 限制上传文件类型与大小。
- 防止目录穿越。
- 防止任意文件读取。
- 防止 SQL 注入。
- Prisma 查询仍要验证用户输入。
- 配置 CORS。
- 限制登录、上传和导出接口频率。
- 日志中不得记录密码、Token 或数据库连接字符串。

- 使用清晰的 service、repository 和 adapter 分层。
- 单个文件职责明确。
- 核心算法必须有注释和测试。
- 所有环境变量提供 .env.example。
- 提供数据库迁移。
- 提供 README，包括安装、配置和启动命令。
- 所有新增功能必须能编译、通过 lint 和测试。

十九、实施方式

请先执行以下步骤，不要直接生成大量未经验证的代码：

1. 分析需求和技术风险。
2. 输出第一阶段架构设计。
3. 确定目录结构。
4. 定义共享数据模型。
5. 定义 Prisma 数据模型。
6. 定义 API 契约。
7. 设计 PixiJS 场景树和相机模型。
8. 给出第一阶段任务清单。
9. 创建项目骨架。
10. 每完成一个模块就运行类型检查、测试和构建。
11. 发现需求冲突时优先保证数据安全、编辑稳定性和大地图扩展能力。
12. 不要未经说明引入新的重量级框架。
