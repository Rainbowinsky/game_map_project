# P12 性能基准运行与记录规范

## 固定场景

`apps/web/src/performance/stamp-benchmark.ts` 生成两组完全确定性的图章数据：2,000 与 5,000 个对象。二者均复用内置的三张 SVG 图章纹理，位置、旋转、缩放、翻转、随机种子与 UUID 均固定；5,000 是第一阶段的压力观察值，不代表十万对象目标。

浏览器验收将所有图章置于一个 Chunk 中，以避免把网络请求数量混入渲染测量。场景加载后记录：

| 字段                                 | 来源                                                             |
| ------------------------------------ | ---------------------------------------------------------------- |
| 浏览器、版本、操作系统、显示器分辨率 | 执行人手工记录                                                   |
| 图章总数与可见数                     | 编辑器状态栏和 Playwright 附件                                   |
| 初始投影耗时                         | Playwright `stamp-benchmark-*.json` 附件中的 `elapsedMs`         |
| 稳态 FPS                             | 编辑器状态栏和附件中的 `rendererFps`                             |
| JS 堆观察值                          | Chromium 可用时附件中的 `usedJsHeapBytes`；不可用时记录为 `null` |
| 图形内存                             | 浏览器性能面板/系统 GPU 面板人工观察，不从 JS 猜测               |

## 自动化运行

```powershell
pnpm --filter @fantasy-map/web test:benchmark
pnpm test:e2e
```

前一条命令给出 2,000/5,000 图章的 Node/Pixi 投影与裁剪吞吐量；后一条命令在 Chromium 实际加载相同场景，并把每组的 JSON 记录附加到 Playwright HTML 报告。基准不使用固定毫秒或 FPS 阈值，因为 CI、GPU 驱动和节能策略会使这些数值失真；它验证场景、可见对象计数、纹理复用与观测数据始终可重复取得。

第二阶段额外提供 500 和 2,000 个混合 `terrain-stroke`、`path`、`region` 对象的确定性缩略导航投影场景。它们只计算低分辨率概览坐标与当前可视矩形，不调用整图 PNG 导出或读取 GPU 像素；同一 `test:benchmark` 命令会同时运行图章和缩略导航场景。结果仍只用于发布候选之间的可复现比较，不设置跨机器 FPS 或耗时阈值。

## 人工验收步骤

1. 以目标发布机的生产构建运行编辑器，执行 `pnpm test:e2e`，在报告中保存两个 `stamp-benchmark-*.json` 附件。
2. 记下设备、浏览器和显示器信息；打开性能面板，确认图章纹理只对应三种内置资源，平移/缩放期间没有 React 主树逐帧提交。
3. 分别平移、缩放、框选一个 2,000 与一个 5,000 场景，记录稳态 FPS、可见/总对象数和峰值 JS/GPU 内存。
4. 将数值与上一个发布候选比较；若出现明显回归，保留 HTML 报告和性能 trace 后再决定是否发布。

第一阶段仅承诺常见桌面设备上空画布接近 60 FPS，以及 2,000 个同纹理图章可继续平移、缩放和选择。十万对象、Chunk 流式加载、空间索引、LOD、Worker 和高分辨率分块导出均属于后续阶段，不能用本基准的结果暗示已经支持。
